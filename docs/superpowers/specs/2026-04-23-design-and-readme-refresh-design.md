# 设计外观与 README 展示重构设计

## 概述

本次重构不改产品功能，不改信息架构的核心职责，目标是把当前界面从“偏 SaaS 模板感的工作台”重构为“有排版感、有温度、适合长期使用的私人判断工作台”。

用户已经明确选择：

- 视觉方向为 `Desk Almanac + Field Notebook` 混合
- 气质优先级为“更有人味”
- 人味主要来自“排版和纸感”，而不是夸张色彩、装饰插画或动效

因此本次重构的核心判断是：

- 以 `Desk Almanac` 的温度、纸感、阅读节奏为主
- 借 `Field Notebook` 的结构骨架和信息秩序为辅
- 保留工具的清晰度，不把产品做成“设计作品先于使用”的样子

一句话目标：

> 让 `oh_my_todo` 看起来像一本被认真排版过的私人面试工作日志，而不是一个通用的白卡片管理面板。

## 设计目标

### 1. 建立自己的视觉语言

当前页面的信息组织已经基本成立，但整体气质仍接近常见的 React SaaS 工作台：

- 深色 marketing hero
- 玻璃感白卡片
- 比较常见的 dashboard 阴影和按钮语言

重构后要让用户第一眼感受到：

- 这是一个“私人判断工具”，不是“团队管理后台”
- 它安静、耐看、有作者感，但不花哨
- 它更像长期使用的桌面工作日志，而不是用来展示 KPI 的 dashboard

### 2. 提升日常使用时的阅读节奏

产品核心任务不是“完成任务”，而是“持续判断”。因此视觉重构要强化：

- 打开页面后先扫未来安排
- 再进入当前推进中的公司判断
- 最后再看 offer 对比和归档

整页应更像连续阅读的工作台，而不是若干功能块拼接。

### 3. 统一产品界面与 README 展示资产

README 目前已经有基础展示资产，但还更像“仓库介绍页”而不是“一个有明确设计语言的项目页”。

本次重构后：

- README 的 banner、动态图和说明排版必须与产品界面同语系
- 用户在 GitHub 首页看到的项目气质，应和打开应用后的气质一致
- 展示图不再只承担“有截图”职责，而要真正表达这个工具为何不同于普通 todo app

## 非目标

- 不新增任何产品功能
- 不调整现有数据模型
- 不改变 `未来 7 天 / 活跃流程 / Offer 对比 / 归档` 这四大信息分区的职责
- 不引入复杂插画、拟物贴纸、重纹理背景、复古装饰元素
- 不为了“更有设计感”而牺牲录入效率和可读性

## 选定方向

### 主方向：Desk Almanac

负责以下特征：

- 暖纸色背景
- 衬线标题
- 更像出版物的留白与边线
- 更安静的节奏
- 带一点私人工作日志的气质

### 辅方向：Field Notebook

负责以下特征：

- 更清楚的分区骨架
- 更明确的分组标题与计数
- 更有秩序的目录感
- 更克制的工具性表达

### 不采用的方向

- 不走冷峻科技 dashboard
- 不走极简黑白艺术作品路线
- 不走夸张 editorial 实验排版

原因是这些方向都会削弱“长期使用的私人工作台”这一核心定位。

## 视觉系统

### 1. 色彩

整体配色改为暖中性纸张体系，避免当前偏蓝灰的 SaaS 气质。

基础色板：

- `--bg-page: #f2eadf`
- `--bg-panel: #f7f1e7`
- `--bg-panel-strong: #efe6d6`
- `--ink-strong: #2b241f`
- `--ink-soft: #5d5147`
- `--line-subtle: rgba(68, 53, 39, 0.14)`
- `--line-strong: rgba(52, 40, 29, 0.22)`
- `--accent-olive: #5f6b5c`
- `--accent-umber: #8a6a4a`
- `--accent-ink: #2f241d`

使用规则：

- 页面底色以纸张暖白为主，不再使用冷蓝灰渐变铺满整页
- 强强调只允许出现在少数关键位置，如主标题、主按钮、关键判断块
- 不做大面积高饱和色块
- 不使用紫色、霓虹色或过强品牌感色彩

### 2. 字体

字体需要承担“人味主要来自排版”的职责，因此必须明确区分 display 与正文。

字体组合：

- Display Serif：`Newsreader`
- Body Sans：`IBM Plex Sans`
- Mono / label 辅助：`IBM Plex Mono`

使用规则：

- 页面主标题、区块标题、关键判断句使用 serif
- 表单、按钮、时间、说明文字、badge 使用 sans
- 少量索引、日期、标签可使用 mono 或高字距 sans
- 不允许整页全部使用 sans，也不允许整页都使用 serif

### 3. 线条、阴影与表面

视觉重心从“阴影漂浮”改为“纸面分层”。

规则：

- 阴影整体减弱，避免 SaaS 卡片悬浮感
- 主要依靠细边线、微弱深浅差和局部深色块建立层次
- 卡片圆角保留，但收敛为更像纸张盒页，而不是玻璃卡片
- 不使用大面积 backdrop blur 和玻璃材质语言

### 4. 排版节奏

页面应更像连续阅读的文档，而不是 widget 容器集合。

规则：

- 更大的标题留白，更紧凑的正文节奏
- 区块内使用“标题 - 说明 - 主内容”的固定秩序
- 优先用对齐、线条、字重差建立层次，而不是靠背景色块堆层
- 保留少量边注、索引标签、计数器，提升“档案/日志”感

## 页面结构重构

### 1. 顶部区域：从 Hero 改为 Masthead

当前顶部过于像营销页 hero，会压过工作台本身。

重构后：

- 移除当前大面积深色渐变 hero
- 改为更像工作日志首页的 `masthead`
- 左侧为一句核心定位和简短说明
- 右侧为本地优先状态、数据操作入口和工作台概览
- “清空本地数据”弱化为次级危险操作，不占据主视觉

目标效果：

- 页面一打开就像“进入你的工作台”
- 不是“先看品牌展示，再进入工具”

### 2. 未来 7 天：做成真正的 Agenda 区

`UpcomingTimeline` 应成为页面第一块有分量的信息区。

重构方向：

- 更像 agenda / running list，而不是一组普通摘要卡片
- 强化日期、公司名、轮次和时间之间的阅读优先级
- 允许少量“日志感”细节，如日期索引、日程密度提示
- 保持快速扫读，不加入任何额外功能

目标效果：

- 用户打开后先迅速知道接下来几天发生什么
- 它像工作台页眉的一部分，不像独立营销模块

### 3. 活跃流程工作区：成为整页中心

这是重构的核心区域。

重构方向：

- 分组容器更像 dossier / archive section，而不是大白卡套小白卡
- 每个 group 的标题、数量、说明更清晰
- 分组切换 tabs 更像索引切换，而不是胶囊按钮组件
- 整体更强调“阅读与维护”，弱化“组件展示”

目标效果：

- 用户能明显感到这里是页面的主工作区
- 信息秩序清楚，但不冰冷

### 4. Offer Comparison：后收但不消失

`OfferComparisonPanel` 仍是核心功能，但视觉上不应和主工作区抢戏。

重构方向：

- 位置仍保留在主页面下半部
- 视觉权重低于活跃流程区
- 表格、对比字段、关键差异应更像“比较页”而不是“面板卡片”

目标效果：

- 用户在需要比较时能进入理性模式
- 但日常首页浏览时，它不会抢走工作区的主导权

### 5. 空状态：像一本新开的工作日志

当前空状态功能上够用，但视觉语气仍偏通用 onboarding。

重构方向：

- 文案更像“开始建立你的判断台”
- 版式更像扉页或首页
- 通过排版和纸感建立期待，而不是靠功能按钮主导

## 组件级重构

### 1. 按钮

当前按钮偏 SaaS CTA 风格，需要收敛。

重构规则：

- 主按钮保留明确性，但不再使用厚重渐变和强阴影
- 次按钮更像工具动作，而不是营销 call-to-action
- ghost 按钮更薄、更安静
- 危险按钮只在明确危险区出现，不抢视觉

### 2. Tabs 与 Badge

当前 tabs 和 badge 组件感较重，需要向“索引”和“编目标签”靠拢。

重构规则：

- tabs 像目录索引，而不是圆角 pill
- badge 像分类标签、状态索引或页边标记
- 减少纯装饰性底色，更多依靠线条、字距、大小写和位置传达层级

### 3. CompanyCard

`CompanyCard` 是本次视觉重构最关键的单元。

目标语义：

- 它不是“功能卡片”
- 它是一页公司的判断档案

结构重构：

- 公司名、岗位名、公司类型、整体判断放在更像页面抬头的位置
- `Company Judgment` 更像摘要段落
- `Interview Schedule` 更像连续记录，而不是表单块堆叠
- `Negotiation` 更像附录或版本记录

视觉重构：

- 减少模块之间的视觉切裂
- 增加纵向阅读连续性
- 让每家公司真的像“一页资料”

### 4. 表单控件

当前输入框与 textarea 的视觉语言过于接近业务系统默认样式。

重构规则：

- 输入框边界更薄，更轻
- 焦点态克制，不做蓝色系统高亮式强反馈
- `datetime-local`、textarea、text input 统一语言
- 保证录入时的光标、边距、行高都更舒服，适合连续编辑

### 5. 指标摘要与统计

顶部和分组内的数字摘要不应像 KPI 卡片。

重构规则：

- 保留数量信息，但避免 dashboard 指标味道
- 更像目录页上的统计摘要
- 数字仍清楚，但不做视觉炫技

## README 展示资产重构

### 1. 总目标

README 不只是告诉别人“这个项目是什么”，还要在第一屏展示它为什么有独特产品判断。

重构后应形成三层表达：

- 第一眼：这是一个有设计语言的项目
- 第二眼：这是给面试判断而不是任务打勾设计的工具
- 第三眼：产品界面和 README 来自同一套视觉系统

### 2. Banner

当前 banner 需要升级为更接近产品主视觉的资产。

要求：

- 使用与产品一致的纸感、排版和结构线条
- 不再强调 terminal gimmick
- 更像一本工作日志的封面或扉页
- 需要能独立表达“interview workbench, not just todo”

### 3. 主展示图 / 动图

README 需要一张真正说明界面气质的主展示图。

要求：

- 展示重构后的真实界面，而不是概念化装饰图
- 优先表现 `未来 7 天 + 活跃流程工作区 + 档案式公司卡片`
- README 保留 1 张主展示静图；如需补充动态展示，只允许额外放 1 张服务于“工作台阅读节奏”的 GIF，不做炫效果动图

### 4. 文案排版

README 的正文组织也需要配合新的设计方向。

要求：

- 更克制地使用 badge
- 更强调项目哲学与使用心智
- 图文顺序与产品信息结构保持一致

## 实现边界与顺序

### 第一阶段：产品界面语言统一

- 重写全局色板和排版变量
- 重做顶部 masthead
- 重做主要 panel、button、tabs、badge 基础样式

### 第二阶段：核心工作区重构

- 重构 `UpcomingTimeline`
- 重构 `CompanyBoard`
- 重构 `CompanyCard`
- 调整 `OfferComparisonPanel` 的视觉权重

### 第三阶段：README 资产同步

- 重做 README banner
- 更新 README 主展示图或动图
- 调整 README 排版和项目说明顺序

## 验收标准

### 视觉验收

- 第一眼不再像通用 SaaS dashboard
- 整体有纸感、排版感和私人工作台气质
- 有温度，但不花哨
- 产品主界面与 README 展示资产风格一致

### 使用体验验收

- 页面打开后阅读顺序清楚：先 agenda，再主工作区，再 comparison
- 长时间看卡片和表单不会觉得视觉疲劳
- 操作入口清楚，但不喧宾夺主

### 范围验收

- 功能行为与现状一致
- 数据模型与交互逻辑不因视觉改造而变化
- README 资产能明确展示产品定位升级

## 风险与约束

### 1. 过度设计风险

如果 serif、纸感、深色块使用过多，界面容易变成“设计作品”而不是“工作工具”。

规避方式：

- serif 只用于标题和关键判断
- 暖色只做底，不做大面积情绪化渲染
- 深色块只用于少量关键强调位

### 2. 信息密度被温柔化风险

如果过分追求“安静”，会削弱工作台的结构清晰度。

规避方式：

- 借 `Field Notebook` 的分组骨架
- 保留清楚的分区标题、计数和索引
- 让工具结构始终先于装饰

### 3. README 和产品脱节风险

如果 README 仍沿用旧的展示图，产品和仓库首页会像两个项目。

规避方式：

- README 资产必须在产品主界面定稿后同步重做
- 所有展示图优先来自真实界面，不做脱离产品的概念图
