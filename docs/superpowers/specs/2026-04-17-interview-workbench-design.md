# 面试工作台 PRD

日期：2026-04-17
主题：面向未来 Todo 产品的面试管理视图原型
状态：已在对话中确认设计方向，待用户审阅文档

## 1. 项目背景

本次要验证的不是一个完整的 Todo 产品，而是一个用于管理面试流程的页面原型。

当前常见做法通常把信息拆散在多个工具里：

- 日历里看已安排的面试时间
- Todo 工具里记要跟进的事项
- 笔记工具里写公司判断和面试感受

这会带来两个问题：

- 打开任意一个工具时，都很难一眼看清“最近到底要处理什么”
- 流程进展和主观判断被拆开后，很难在一个上下文里做决策

因此，本原型要验证一种新的页面形态：把`已确定的面试时间`、`按公司组织的流程管理`、`对公司的主观感受与判断`放进同一个工作台里。

## 2. 产品定位

第一阶段产品定位为：`面试管理视图原型`。

它不是通用 Todo 产品，也不是完整的多端同步系统，而是一个用于验证信息架构和工作流是否成立的桌面 Web 原型。

本期优先验证 3 件事，优先级如下：

1. 用户打开页面后，能立刻知道今天或接下来最近的面试是什么。
2. 以分组视图管理公司流程时，是否比普通列表或笔记式记录更顺手。
3. 把流程信息和主观感受放在同一页面里，是否比“笔记 + 日历 + Todo”分散管理更好用。

## 3. 本期目标

- 让用户打开页面后，在几秒内看清未来 7 天内已确定的面试安排。
- 让用户在一个页面内管理多个活跃公司的面试流程。
- 让用户在公司维度和轮次维度都能记录感受与判断。
- 验证“创业公司 / 大厂”是否适合作为默认分组视图。
- 同时保留未来切换其他分组方式的能力，不把当前视图写死成唯一结构。
- 在不引入自动采集、同步、多端适配的前提下，先验证页面排布和交互工作流。

## 4. 非目标

以下能力不纳入第一阶段原型范围：

- 从日历、邮件、剪贴板、ATS 自动导入面试信息
- 自动生成待办、提醒或跟进任务
- 移动端界面适配
- 实时多端同步
- 通用 Todo 产品的信息架构设计

## 5. 范围定义

第一阶段范围明确如下：

- 仅支持`桌面 Web`
- 仅支持`纯手动录入`
- 聚焦一个核心页面：`面试工作台`
- 支持活跃流程管理与归档流程回看

虽然第一阶段不做多端和同步，但本期设计必须为未来扩展留下空间。因此不能把：

- `创业公司 / 大厂` 写死成唯一结构
- 页面分组逻辑写死在 UI 上
- 公司、流程、轮次混成一个层级

## 6. 核心方案结论

本次确认采用的方案是：`可切换分组的工作台`

选择这个方案的原因：

- 它保留了你最初的直觉：页面默认可以用`创业公司 / 大厂`双列来展示。
- 它把这个双列定义为`默认视图`，而不是产品唯一结构。
- 它允许后续继续验证其他分组方式，而不需要推翻整页设计。
- 它比主从式详情布局保留更多全局感，更符合“打开后先看最近面试、再看整体推进情况”的核心诉求。

因此，页面默认采用：

- `按公司类型分组`
- 左列为`创业公司`
- 右列为`大厂`

但这只是默认打开方式，不是永久产品模型。

## 7. 页面信息架构

页面分为 3 个主区域。

### 7.1 顶部时间区

职责：
只展示`已确定时间`的面试。

规则：

- 展示范围为`今天起未来 7 天`
- 只展示已经确认日期和时间的面试
- 严格按时间升序排列
- 展示单位是`面试事件`，不是公司摘要
- 不展示准备动作、等待结果、催进度、发感谢信等非面试事件

这个区域只回答一个问题：
`接下来已经确定会发生哪些面试？`

### 7.2 中部工作台

职责：
管理所有`仍在推进中`的公司与流程。

规则：

- 这是页面的主工作区
- 默认分组方式为`公司类型`
- 分组方式必须可切换
- 第一版建议支持 3 种分组方式：
  - `公司类型`
  - `流程阶段`
  - `个人优先级`
- 这里只展示活跃流程
- 只要某家公司仍有至少一个活跃流程，它就应出现在主工作区
- 该区域既要支持浏览，也要支持直接编辑

这个区域回答的问题是：
`当前有哪些公司在推进？我现在该更新什么？`

### 7.3 底部归档区

职责：
承接已结束、已搁置或长期无动静的流程，避免干扰主工作区。

规则：

- 包含已拒绝、已放弃、长期无推进的流程
- 与主工作区明显分隔
- 默认折叠或弱化展示
- 只有当某家公司下的所有流程都已归档或结束时，这家公司才整体进入归档区

这个区域回答的问题是：
`哪些流程结束了，未来如需回看判断依据，可以去哪里找？`

## 8. 数据模型

第一阶段原型建议采用 4 层结构。

### 8.1 公司

公司是工作台里的父层对象。

公司级字段：

- 公司名称
- 公司类型
- 整体印象
- 亮点
- 风险点
- 个人意向或优先级

### 8.2 流程

一个公司下可以有多个流程。

典型场景：

- 同一家公司有多个岗位
- 同一家公司存在多条招聘流程

流程级字段：

- 岗位名称
- 当前阶段
- 下一步信息
- 流程整体状态
- 归档状态

### 8.3 轮次

每条流程下有多个轮次记录。

轮次级字段：

- 轮次名称
- 时间，可为空，只有确认后才写入
- 轮次状态
- 轮次备注

第一版推荐支持的轮次状态：

- `待约面`
- `已安排`
- `已完成`
- `等待结果`
- `已结束`

### 8.4 视图分组

分组是视图层元数据，不是记录本体。

例如：

- 在`公司类型`视图中，公司可能出现在`创业公司`
- 在`流程阶段`视图中，同一家公司可能出现在`面试中`
- 在`个人优先级`视图中，同一家公司可能出现在`高优先级`

这个约束很重要，因为未来如果要做移动端或同步，不应该把当前双列排布当成底层数据模型。

## 9. 公司卡设计

公司卡是整个页面的核心交互单元。

### 9.1 默认收起态

每张公司卡在默认收起时，应同时呈现流程信息和主观判断，也就是`混合摘要`。

默认应展示的信息：

- 公司名称
- 当前所在分组标签
- 流程摘要：当前到第几轮、最近一次进展、下一场已定面试时间
- 判断摘要：整体意向、亮点、风险点
- 快捷操作入口

第一版快捷操作建议包括：

- `新增轮次`
- `记录感受`
- `改状态`
- `归档`

默认收起态必须帮助用户在不展开的情况下回答两个问题：

- 这家公司当前推进到哪里了
- 这家公司现在值不值得继续投入精力

### 9.2 展开态

展开后的公司卡分为两个层级：

第一层：`公司级总览`

- 整体印象
- 团队或文化判断
- 岗位吸引力
- 主要风险
- 你的阶段性结论

第二层：`流程列表`

- 同一家公司下可展示多个流程
- 每条流程下继续展示轮次记录
- 每个轮次都可记录时间、状态和备注

### 9.3 轮次记录

第一版轮次记录应尽量轻量，但足以支撑真实使用。

每条轮次至少包含：

- 轮次名称
- 时间
- 状态
- 备注

备注内容可包含：

- 面试官印象
- 自我发挥评价
- 异常信号
- 对下一步的判断

## 10. 交互原则

这个页面不是纯浏览页，而是`工作台`，所以必须支持高频直接编辑。

第一版交互原则如下：

- 常见编辑动作尽量在当前页原地完成
- 用户不应为了常规更新频繁跳转详情页
- 小型状态修改使用轻量交互
- 较长的文本感受可在卡片内展开为更大的输入区域

第一版重点支持的高频动作：

- 新增一轮面试
- 修改流程状态
- 设置或修改已确认面试时间
- 写公司级感受
- 写轮次级备注

## 11. 展示规则

为避免重复和噪音，页面必须遵守以下规则：

- 只有未来 7 天内且时间已确认的面试，才进入顶部时间区
- 没有确定时间的流程，绝不进入顶部时间区
- 顶部时间区按`面试事件`展示，而不是按公司聚合
- 中部工作台展示活跃公司卡
- 只要仍在推进，即使没有已确定时间，公司也可以出现在中部工作台
- 底部归档区只承接结束或停滞流程，不应与主工作区竞争视觉注意力
- 只有当公司下所有流程都归档或结束时，公司才整体进入归档区

## 12. 原型成功标准

这个原型成立，必须同时满足`主观成功`和`行为成功`两类标准。

### 12.1 主观成功

在连续使用 1 到 2 周后，用户主观上愿意继续使用这个工作台维护面试信息，而不是回到原来的笔记 + Todo 组合。

### 12.2 行为成功

用户应能够做到：

- 打开页面后 10 秒内看清未来 7 天的已定面试
- 30 秒内完成一家公司流程状态更新
- 1 分钟内补完一轮面试的时间、状态和备注
- 在多家公司并行推进时，不需要频繁切换到其他工具查看上下文

## 13. 未来扩展约束

虽然本期只做桌面 Web 手动录入原型，但设计不能阻碍后续扩展。

未来可能扩展的方向包括：

- 移动端适配
- 多端同步
- 更完整的待办与提醒能力
- 更多分组方式
- 与日历、邮件、消息来源的接入

因此需要明确几条约束：

- 分组逻辑不能成为底层唯一结构
- 公司、流程、轮次必须保持清晰分层
- 顶部时间区的面试事件应当由轮次时间推导出来，而不是单独建一套割裂的数据系统

## 14. 被否决的方案

### 14.1 固定双列方案

即页面永久固定为`创业公司 / 大厂`两列。

否决原因：

- 虽然直观，但会把当前实验性视图误写成永久产品结构
- 后续一旦要按阶段、优先级切换，就会推翻现有实现

### 14.2 主从详情工作台

即左边列表、右边详情编辑的主从结构。

否决原因：

- 它更适合深度编辑
- 但会削弱“打开后一眼看清全局”的能力
- 与本期最高优先级验证目标不完全一致

### 14.3 一上来就做完整 Todo 产品

否决原因：

- 会把太多问题混在一起：面试视图、通用任务、多端、同步、采集链路
- 会显著增加 PRD 范围，导致第一阶段难以验证真正关键的问题

## 15. 最终结论

第一阶段应构建一个`桌面 Web、纯手动录入`的面试工作台原型。

页面结构明确为：

- 顶部：未来 7 天已确定时间的面试时间区
- 中部：支持可切换分组的活跃公司工作台，默认按公司类型分组
- 底部：归档区

核心交互对象是`公司卡`，公司卡同时承载：

- 公司级判断
- 流程级进展
- 轮次级记录

`创业公司 / 大厂`应被定义为默认视图，而不是永久产品模型。
