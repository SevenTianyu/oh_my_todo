# 谈薪模块与 Offer 对比设计

## 概述

当前工作台已经覆盖两类核心信息：

- 公司判断
- 面试安排与轮次备注

但当用户完成前面几轮面试、进入 HR 谈薪与最终决策阶段时，现有模型会出现两个问题：

1. “谈薪”既不是普通面试轮次，也不只是备注，硬塞进现有面试模块会让信息结构失真
2. 用户真正需要的不是完整复刻谈薪过程，而是把不同公司的最新方案标准化后做横向比较

本次设计将新增一个独立于“公司判断 / 面试安排”的第三模块：`谈薪`，并补充一个独立的 `Offer 对比` 视图。整体原则是：

- 谈薪记录保持轻量，不做成重流程编辑器
- 每次修改都保留历史版本，支持回看谈薪变化
- 总包不是用户手填，而是系统根据原始字段自动计算
- 产品核心不是“记下谈薪过程”，而是“帮助用户比较并做最终判断”

## 目标

### 1. 把谈薪从面试模块中拆出来

- 公司卡片新增与“公司判断”“面试安排”并列的第三块：`谈薪`
- 谈薪模块不复用面试轮次编辑器，不要求用户逐轮维护复杂流程
- 进入谈薪后，公司在主工作区里的状态应与面试中区分开

### 2. 让最终比较成为一等公民

- 新增独立的 `Offer 对比` 视图
- 默认只展示当前 `谈薪中` 的公司
- 支持切换到“全部有薪资记录的公司”，保留历史比较能力

### 3. 保留多轮谈薪变化，但压低录入成本

- 谈薪模块采用“版本快照”而不是“轮次编辑器”
- 每次保存谈薪信息都新增一条版本记录，不覆盖旧值
- 公司卡片默认只展示最新方案摘要和历史版本列表

## 非目标

- 不把谈薪做成与面试模块等价的多轮流程管理器
- 不在首版支持复杂的 equity 建模，例如不同年份归属比例、 cliff、双触发条款等
- 不在首版引入自动接受 offer、自动归档、自动结束流程
- 不在首版引入多维排序、权重评分、决策打分器等高级比较能力

## 产品结构

### 1. 公司卡片新增“谈薪”模块

每张公司卡片保留三个并列模块：

- 公司判断
- 面试安排
- 谈薪

谈薪模块职责：

- 展示当前谈薪状态
- 展示最新一版谈薪摘要
- 展示历次谈薪版本变化
- 提供编辑入口与进入 `Offer 对比` 的入口

谈薪模块不负责承载完整横向比较信息。卡片内只展示决策所需的轻摘要。

### 2. 首页阶段分组新增“谈薪中”

当前首页分组从：

- 筛选中
- 面试中

扩展为：

- 筛选中
- 面试中
- 谈薪中

当公司确认进入谈薪后，应优先归入 `谈薪中`，而不是继续停留在 `面试中`。

### 3. 新增独立视图“Offer 对比”

`Offer 对比` 视图作为面向最终决策的专用面板，职责是：

- 将不同公司的最新谈薪方案标准化展示
- 展示系统计算后的首年总包与长期年化总包
- 支持在当前谈薪公司之间做横向比较
- 支持查看历史有薪资记录的公司

默认筛选规则：

- 默认展示：`谈薪中`
- 可切换到：全部有薪资记录的公司

## 数据模型设计

### 1. 谈薪数据挂载在公司层

谈薪模块应挂载在 `CompanyRecord` 层，而不是直接塞进 `InterviewProcess` 或 `RoundRecord`。

原因：

- UI 上谈薪是公司卡片的第三块，更符合公司级体验
- 用户对谈薪的认知是“这家公司进入了谈薪阶段”，而不是“某一轮面试继续推进”
- 后续对比视图本身也是公司维度，而不是单轮维度

### 2. 谈薪记录必须保留来源岗位

虽然挂载在公司层，但谈薪实体必须包含 `sourceProcessId`，用于指向它是从哪个岗位流程走出来的。

这保证：

- 卡片上能明确看到这份谈薪来自哪个 role
- 比较视图不丢失岗位语境
- 后续若同一公司出现多个岗位进入谈薪，仍有扩展空间

### 3. 推荐的数据结构

建议在 `CompanyRecord` 下新增一块独立数据，例如：

```ts
type NegotiationStatus =
  | "inactive"
  | "active"
  | "accepted"
  | "declined"
  | "terminated";

interface NegotiationSnapshot {
  id: string;
  version: number;
  createdAt: string;
  title: string;
  level: string;
  city: string;
  workMode: string;
  baseMonthlySalary: number | null;
  salaryMonths: number | null;
  annualBonusCash: number | null;
  signOnBonus: number | null;
  relocationBonus: number | null;
  equityShares: number | null;
  equityStrikePrice: number | null;
  equityReferencePrice: number | null;
  equityVestingYears: number | null;
  deadline: string | null;
  hrSignal: string;
  notes: string;
}

interface CompensationNegotiation {
  status: NegotiationStatus;
  sourceProcessId: string | null;
  startedAt: string | null;
  endedAt: string | null;
  latestSnapshotId: string | null;
  snapshots: NegotiationSnapshot[];
}
```

其中：

- `inactive` 表示尚未进入谈薪
- `active` 表示谈薪中
- `accepted / declined / terminated` 表示谈薪已结束且结果已明确

### 4. 数据兼容

- 旧快照导入时，不要求已有 `negotiation` 字段
- 缺失时按 `inactive + 空 snapshots` 处理
- 新快照导出时应包含谈薪数据结构

## 状态流转

### 1. 激活方式：系统建议 + 用户确认

谈薪模块不应完全自动开启，而应采用“建议 + 确认”模式。

系统可以根据以下信号给出建议：

- 活跃流程已经存在多轮完成或等待结果
- `nextStep` 呈现明显的 HR / offer / compensation 语义
- 当前流程已经缺少明确的后续技术/业务面试安排

但真正切到 `谈薪中` 必须由用户确认一次。

目标是避免误判：

- 普通 HR follow-up
- 结果通知
- 背景调查
- 泛化的收尾沟通

这些都不应被系统直接认定为谈薪。

### 2. 首页阶段判定优先级

建议公司的主阶段判定优先级为：

1. 谈薪中
2. 面试中
3. 筛选中

只要公司谈薪状态为 `active`，该公司在主工作区中应归入 `谈薪中`。

### 3. 谈薪结束状态

谈薪结束后不直接等同于归档，而先记录结果：

- `已接受`
- `已拒绝`
- `已终止`

其中：

- `已接受`：用户接受该方案
- `已拒绝`：用户主动拒绝该方案
- `已终止`：谈崩、HC 关闭、流程拖没、对方撤回等

是否进一步进入归档区，仍由现有流程归档逻辑控制；谈薪结果与流程归档是两个相邻但不等价的概念。

## 谈薪模块交互

### 1. 模块默认态

当谈薪未开启时，模块展示：

- 当前未进入谈薪
- 若系统已检测到信号，展示“建议进入谈薪”的提示
- 用户可手动确认进入谈薪

### 2. 激活后展示

谈薪激活后，模块展示：

- 当前状态：谈薪中 / 已接受 / 已拒绝 / 已终止
- 来源岗位流程
- 最近更新时间
- 最新一版谈薪摘要
- 历次版本列表

### 3. 编辑方式

谈薪模块采用单表单编辑：

- 用户打开当前谈薪表单
- 修改最新已知方案
- 保存后生成一条新的 `snapshot`

旧版本不被覆盖，只做历史记录。

这意味着谈薪体验更像“记录新报价版本”，而不是“维护一个复杂流程图”。

## 谈薪记录字段

首版建议保留以下字段，足以支撑比较，但不过度建模：

- `title`
- `level`
- `city`
- `workMode`
- `baseMonthlySalary`
- `salaryMonths`
- `annualBonusCash`
- `signOnBonus`
- `relocationBonus`
- `equityShares`
- `equityStrikePrice`
- `equityReferencePrice`
- `equityVestingYears`
- `deadline`
- `hrSignal`
- `notes`

字段意图：

- `title / level / city / workMode` 负责上下文比较
- `baseMonthlySalary / salaryMonths / annualBonusCash` 负责现金部分
- `signOnBonus / relocationBonus` 负责一次性补贴
- `equity*` 负责系统计算 equity 年化值
- `deadline / hrSignal / notes` 负责沟通与决策上下文

## 计算规则

### 1. 总包是系统计算值，不可手填

用户不直接输入“总包”。比较页中的总包数字全部来自系统计算。

### 2. 现金计算

基于最新一版 `snapshot` 计算：

- `年固定现金 = baseMonthlySalary × salaryMonths`
- `年现金合计 = 年固定现金 + annualBonusCash + signOnBonus + relocationBonus`

首版默认把 `signOnBonus / relocationBonus` 计入首年比较，但不计入长期年化比较。

### 3. Equity 计算

首版采用统一近似规则：

- `equity总估值 = (equityReferencePrice - equityStrikePrice) × equityShares`
- 若为 RSU，可视作 `equityStrikePrice = 0`
- 若任一关键字段缺失，则 equity 相关计算结果为空，不强行补零伪装为完整数据
- `equity年化值 = equity总估值 ÷ equityVestingYears`

### 4. 两个核心总包指标

比较页应至少展示两个标准化结果：

- `首年总包 = 年现金合计 + equity年化值`
- `长期年化总包 = (年固定现金 + annualBonusCash) + equity年化值`

原因：

- 首年总包有助于反映签字费与搬家费对当前决策的影响
- 长期年化总包更适合排除一次性项目带来的失真

### 5. 计算展示原则

- 有值才算，不拿缺失字段凑完整结果
- 缺失时应在 UI 上明显体现“信息不完整”
- 比较页应优先显示“系统已算出的结果”，同时允许用户展开查看原始字段

## Offer 对比视图

### 1. 默认展示范围

- 默认：当前 `谈薪中` 的公司
- 可切换：全部有薪资记录的公司

### 2. 默认列

首版建议展示以下核心列：

- 公司
- 来源岗位
- 最新谈薪轮次
- 职级 / Title
- 城市 / 办公方式
- 月薪 × 月数
- 年度奖金
- 股票/期权年化值
- 首年总包
- 长期年化总包
- HR 当前口径
- 截止时间
- 主观判断 / 是否值得去

### 3. 历史变化查看

比较页中的每家公司应支持展开，查看历次谈薪版本变化，例如：

- 第 1 轮谈薪
- 第 2 轮谈薪
- 第 3 轮谈薪

展开后重点看：

- 现金数字如何变化
- equity 是否改善
- deadline 是否变化
- HR 口径和备注如何变化

## 与现有工作台的关系

### 1. 公司卡片结构变化

从当前两块：

- 公司判断
- 面试安排

变为三块：

- 公司判断
- 面试安排
- 谈薪

### 2. 主工作区分组变化

分组入口保留“按流程阶段分组”，但阶段集合扩展为：

- 筛选中
- 面试中
- 谈薪中

### 3. 现有面试数据不承担谈薪职责

- 面试模块继续维护轮次时间、状态和备注
- 不要求把谈薪细节写进 `round.notes`
- 谈薪是一个新的业务模块，而不是对面试模块的补丁

## 错误处理与边界

### 1. 缺失薪资字段

- 若缺失现金字段，相关总包不计算
- 若缺失 equity 关键字段，equity 年化值不计算
- UI 明确提示“信息不完整”，避免误导比较

### 2. 结束状态与归档不一致

允许存在以下情形：

- 谈薪已结束，但流程尚未归档
- 流程已归档，但历史上存在薪资记录

系统不强行耦合这两类状态。

### 3. 来源流程缺失

若历史数据中 `sourceProcessId` 指向的流程已不存在：

- 不阻塞展示历史谈薪记录
- UI 展示“来源岗位已删除/不可用”

## 测试与验收

### 自动化测试

- 选择器测试：
  - 谈薪中的公司优先分到 `谈薪中`
  - 未谈薪但有推进痕迹的公司仍分到 `面试中`
  - 无推进痕迹且未谈薪的公司分到 `筛选中`
- 组件测试：
  - 公司卡片展示第三块 `谈薪`
  - 保存谈薪信息后生成新的历史版本，而不是覆盖旧版本
  - `Offer 对比` 视图默认只显示 `谈薪中`
  - 切换到“全部有薪资记录的公司”后显示历史记录
- 计算测试：
  - 现金部分按月薪 × 月数 + 奖金正确计算
  - 首年总包和长期年化总包按预期区分一次性补贴
  - equity 缺失关键字段时返回空值而不是错误值
- 存储测试：
  - 导入旧 JSON 时缺失谈薪字段仍成功
  - 导出新 JSON 时包含谈薪结构

### 手工验收

- 当某家公司进入谈薪后，首页主工作区中它被归到 `谈薪中`
- 用户在公司卡片里能看到谈薪模块，不需要把信息塞回面试备注
- 每次调整薪资方案后，历史版本仍然可回看
- 对比页中总包由系统计算得出，而不是手填
- 默认只看当前谈薪公司，但可以切换查看历史有薪资记录的公司

## 方案选择结论

本次最终采用 `比较中心型` 方案：

- 谈薪模块在公司卡片中轻量记录
- 历史版本完整保留
- 首页阶段新增 `谈薪中`
- 决策核心放在独立的 `Offer 对比` 视图

这个方案与用户目标最一致：不把谈薪做成额外负担，而是把它变成最终判断时最有用的结构化信息。
